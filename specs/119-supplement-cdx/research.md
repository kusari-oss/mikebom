# Research — Developer-asserted source-of-truth supplement (v0.1, CDX 1.6 input)

**Feature**: 119-supplement-cdx
**Date**: 2026-06-13
**Status**: Decisions resolved; no NEEDS CLARIFICATION markers remaining.

## Decision 1 — Validation mechanism: hand-rolled structural check

**Decision**: A hand-rolled structural validator in `mikebom-cli/src/supplement/parser.rs` that asserts the supplement file's `bomFormat == "CycloneDX"`, `specVersion ∈ {"1.4", "1.5", "1.6"}`, and that the `components[]` / `services[]` / `dependencies[]` arrays' entries carry the keys mikebom actually consumes (component: `purl` string; service: `name` + `provider.name` strings; dependency edge: `ref` + `dependsOn` array). Do NOT add the `jsonschema` crate as a runtime Cargo dependency.

**Rationale**:
- **Zero new Cargo dependencies** matches mikebom's standing posture across every milestone since 002. The `jsonschema` crate already lives in `[dev-dependencies]` (used by `tests/sbom_user_metadata.rs:1482-1516`); promoting it to a runtime dep would add ~5 MB to the binary + pull in `regex-syntax`, `fancy-regex`, and `uuid` transitively.
- **Mikebom reads a SUBSET of CDX 1.6 from the supplement**: only the fields it actually consumes during merge. A full JSON Schema validation would also check `vulnerabilities[]`, `formulation[]`, `evidence` shapes, and ~50 other CDX 1.6 fields that the merge ignores. Strict validation of fields we don't read invents work the operator hasn't asked for.
- **Testable + stays in sync**: the hand-rolled validator's required-keys list is the SAME list the merge code reads. A future PR that adds a new merged field (e.g., consuming `evidence.identity` from the supplement) extends BOTH the validator and the merge in one diff — the validation can't drift from what the merge actually accepts.
- **Compatibility with earlier CDX versions**: accepting `specVersion ∈ {"1.4", "1.5", "1.6"}` lets operators feed supplement files generated by older CDX-emitting tools (CMake-SBOM-Builder emits 1.5; trivy emits 1.4) without forcing them to upgrade their toolchain. The fields we read (PURL, name, provider, ref, dependsOn) are unchanged across these three versions.

**Alternatives considered**:
- **Add `jsonschema` as a runtime dep + validate against bundled CDX 1.6 schema** — works but rejected per the dep + invented-work objections above. Future v2 layering can swap to this without breaking the v0.1 contract.
- **Accept any JSON; fail at merge time with field-by-field errors** — too permissive; operators get confusing failures from the merge code instead of clear "your supplement is malformed" feedback at scan startup.
- **Validate against a custom mikebom-supplement-v1 schema** that's a subset of CDX 1.6 — over-engineered; the hand-rolled structural check is simpler and covers the same intent.

## Decision 2 — Merge pipeline slot

**Decision**: Insert `supplement::merge(scanner_components, supplement)` at `mikebom-cli/src/generate/cyclonedx/builder.rs:355` — immediately after `build_components()` populates the scanner-discovered set, before `build_compositions()` and `build_dependencies()` run. The merge mutates the components Vec in place and returns a `Vec<ServiceEntry>` that a new `build_services()` function (paralleling `dependencies.rs`) consumes.

**Rationale**:
- **One-point insertion = uniform behavior across formats**: the merged `ResolvedComponent` set + the new `Vec<ServiceEntry>` are inputs to BOTH the CDX builder AND the SPDX 2.3 + SPDX 3 builders. Doing the merge once at the highest-level shared input means SPDX outputs automatically inherit the supplement's components (with the SPDX-side projection of services-as-packages per spec edge case 3).
- **Post-dedup means scanner's authoritative shape wins on identity**: by the time the merge runs, milestone-105's `SourceMechanism` dedup pipeline has already collapsed scanner-side PURL collisions into one entry per canonical PURL. The supplement merge is operating on a deduplicated scanner set, so the merge's PURL exact-match lookup (FR-010) is unambiguous.
- **Pre-emission means a single emission path doesn't need supplement-awareness**: the CDX builder's `build_components()` at builder.rs:558-1016 and the SPDX `build_packages()` at `generate/spdx/document.rs:233` continue to consume `Vec<ResolvedComponent>` the same way they do today. The supplement is invisible to the format-specific emission code.
- **The merge is small and local**: ~120 LoC in `supplement/merge.rs` to do (a) build a `HashMap<Purl, usize>` index over scanner components, (b) for each supplement component, look up the index, (c) if hit, call `conflict::resolve_component()` and replace in place; if miss, push as a new entry with `mikebom:source-tier = declared`.

**Alternatives considered**:
- **Merge at per-format emission time** (i.e., in the CDX builder AND in each SPDX builder separately) — rejected: triples the surface area + invites divergence between formats.
- **Merge as a new `SourceMechanism::Declared` variant in the dedup pipeline at `scan_fs/dedup.rs:58-88`** — works, but conflates "operator authoritative declaration" with "filesystem reader output." The dedup pipeline's mental model is "collapse colliding readers"; supplement is a fundamentally different INPUT SOURCE. Cleaner as a sibling stage.
- **Merge inside `scan_cmd.rs` before invoking the builders** — works structurally but pollutes scan_cmd's already-long function with merge logic. Cleaner as a dedicated module.

## Decision 3 — Field-set membership for FR-006 (scanner wins) and FR-007 (developer wins)

**Decision**: The two field sets are FIXED for v0.1 per spec's clarifications.

**Scanner wins (bytes-derived facts)** — `conflict::SCANNER_AUTHORITATIVE_FIELDS`:
- `hashes[]` (every algorithm — sha256, sha1, etc.)
- `cpe` (single value per component, may be Vec in CDX 1.6)
- canonical `purl` (the field itself; entry IS the join key)
- `version` (when scanner detected via embedded version string or fingerprint)
- `binary_role` (milestone-104 / 099 binary-fingerprint-derived component identity)

**Developer wins (metadata)** — `conflict::DEVELOPER_AUTHORITATIVE_FIELDS`:
- `licenses[]`
- `concluded_licenses[]` (ClearlyDefined-style operator concluded license)
- `supplier`
- `copyright`
- `name` (display name, distinct from PURL identity)
- `description`
- `externalReferences[]` (ALL types — `website` / `documentation` / `distribution` / `vcs` / `mailing-list` / `issue-tracker` / etc.; operator-supplied URLs are operator-domain regardless of reference type)

**Default for fields not in either set**: scanner wins. The safety property in FR-015 ("developer cannot suppress scanner detection of bytes-evident content") motivates this default — when in doubt, preserve scanner emission.

**Rationale**:
- The two lists are taken directly from spec FR-006 + FR-007 (which were already deliberately narrow); this decision just commits to them in the implementation.
- Encoding the lists as `const &[&str]` slices in `conflict.rs` (not Rust enums) lets future operators read the source code to understand the partition without a complex docs-vs-code drift problem.
- The catch-all "scanner wins" default is the conservative choice. Adding a field to `DEVELOPER_AUTHORITATIVE_FIELDS` is a one-line PR; removing one from `SCANNER_AUTHORITATIVE_FIELDS` is also a one-line PR. Both extensions are reviewer-policed contract decisions.

**Alternatives considered**:
- **Operator-tunable via a CLI flag** (e.g., `--supplement-trust-licenses=false`) — rejected as scope creep; the fixed split is the v0.1 contract per spec Assumption #3.
- **Per-component override declared inside the supplement file** (e.g., a `mikebom:trust` property on each declaration) — rejected; same scope-creep objection, and it invites operators to declare per-component "trust" assertions that bypass the safety property.

## Decision 4 — SPDX 2.3 / SPDX 3 services projection

**Decision**:
- **SPDX 2.3**: supplement's `services[]` entries project onto `packages[]` carrying a `mikebom:component-role = "saas-service"` annotation following the milestone-049 C40 pattern. Emission entry point: `mikebom-cli/src/generate/spdx/packages.rs` (existing `build_packages()` is extended to accept a `Vec<ServiceEntry>` parameter).
- **SPDX 3.0.1**: supplement's `services[]` entries emit as SPDX 3 `Service` Elements where supported by the schema; otherwise fall back to the SPDX 2.3 projection pattern (Package + `mikebom:component-role` annotation) wrapped in a `Bundle` per the issue body's research summary. Emission entry point: `mikebom-cli/src/generate/spdx/v3_packages.rs` (or equivalent).

**Rationale**:
- SPDX 2.3 has no native `Service` type. The C40 `mikebom:component-role` pattern is the standing project mechanism for component-role-style classification when no native field exists. Reusing it preserves consumer interoperability — downstream tooling that already understands C40 picks up the new "saas-service" value automatically.
- SPDX 3.0.1's `software_Package` taxonomy is the canonical home for declared services. Where the spec doesn't perfectly fit, fall back to the 2.3 pattern preserves the operator-visible promise that supplement-declared services appear in the SBOM regardless of output format.
- The fallback semantics are documented in spec edge case 3; this decision finalizes the implementation path.

**Alternatives considered**:
- **Drop SPDX support for services in v0.1; emit them only in CDX** — rejected: operators emitting SPDX as their primary output (federal procurement consumers, supply-chain auditors) would lose visibility of declared services. The annotation-based projection is the right interoperability bridge.
- **Define a brand-new SPDX 2.3 annotation key like `mikebom:saas-service`** — rejected: reuses-existing-pattern is better than invent-new-key per Constitution Principle V's "prefer existing pattern" guidance.

## Decision 5 — `mikebom:assertion-conflict` annotation shape

**Decision**: JSON object embedded as the value of a CDX `properties[]` entry (or SPDX `Annotation`) with the canonical structure:

```json
{
  "field": "licenses",
  "scanner_value": [],
  "supplement_value": [{"license": {"id": "Apache-2.0"}}],
  "winner": "supplement",
  "justification": "developer-metadata-override"
}
```

Where:
- `field` is the conflicted CDX field name (string)
- `scanner_value` and `supplement_value` are the JSON values from each side (any JSON type allowed)
- `winner` ∈ `{"scanner", "supplement"}` — the side that ended up emitted
- `justification` ∈ `{"developer-metadata-override", "bytes-evident-detection-preserved"}` per spec FR-009 minimal enum (clarification Q3)

The annotation may appear MULTIPLE TIMES per component (one entry per conflicted field). Consumers reading the SBOM enumerate all conflicts by grepping `properties[].name == "mikebom:assertion-conflict"` (with array iteration since the same name can appear repeatedly).

**Rationale**:
- **Self-contained**: a single annotation entry tells consumers WHAT field conflicted, WHAT the two sides asserted, WHICH won, and WHY. No cross-referencing other annotations required.
- **Machine-parseable**: JSON-encoded values let consumers write structured filters (e.g., "show me every component where the operator declared a license the scanner didn't detect").
- **Justification enum is small and stable**: only two values per spec clarification Q3, so the contract is tight + future expansion is a strict-superset extension that doesn't break existing consumers.

**Alternatives considered**:
- **One annotation per disagreement type** (e.g., `mikebom:license-conflict` + `mikebom:supplier-conflict` + ...) — rejected: explodes the annotation namespace + creates a maintenance burden as the field set grows.
- **Compressed string format** (e.g., `"licenses|scanner=[]|supplement=Apache-2.0|winner=supplement"`) — rejected: harder to parse, harder to extend, harder to validate.

## Decision 6 — `mikebom:supplement-cdx` provenance shape

**Decision**: Document-scope `metadata.properties[]` entry with:

- **name**: `"mikebom:supplement-cdx"`
- **value**: `"<path>@sha256:<hex>"`

Where `<path>` is the verbatim string the operator passed to `--supplement-cdx` (preserved as-is to match what the operator typed; not absolutized or normalized), and `<hex>` is the lowercase hex of the SHA-256 hash of the supplement file's raw bytes.

Examples:
- `supplement.cdx.json@sha256:deadbeef...` (relative path)
- `/etc/mikebom/supplement.cdx.json@sha256:cafebabe...` (absolute path)

**Rationale**:
- **Verbatim path preservation**: operators reading the emitted SBOM see EXACTLY what they typed at the command line — easier for grep / log correlation than a normalized absolute path that doesn't match the script that produced it.
- **SHA-256 cryptographic hash**: lets consumers verify the supplement file's integrity at audit time. If the supplement file is later modified, the SBOM's provenance record diverges from the file's current hash, making the tampering detectable.
- **Document-scope single occurrence**: per FR-001, only ONE supplement file is honored in v0.1 (multi-file deferred to v2). One annotation slot at `metadata.properties[]` is correct.
- **Format matches the milestone-113 `mikebom:exclude-path` precedent at `generate/cyclonedx/metadata.rs:126-135`**: same emission pattern; consumer tools that already parse milestone-113 envelope annotations pick this up automatically.

**Alternatives considered**:
- **Embed the full supplement file content as a base64 blob** — rejected: bloats every SBOM that uses a supplement; the hash-reference pattern is the standing project convention for external-file references.
- **Use multiple property keys** (`mikebom:supplement-cdx-path` + `mikebom:supplement-cdx-sha256`) — rejected: doubles the annotation count for no benefit; combined `path@sha256:hex` is compact and unambiguous.

## Decision 7 — Supplement module layout

**Decision**: New module at `mikebom-cli/src/supplement/` with five files per the plan.md § Source Code layout:
- `mod.rs` — public `Supplement` struct + `pub fn merge()` entry point + module-level documentation
- `parser.rs` — hand-rolled structural CDX 1.6 subset validator (Decision 1)
- `merge.rs` — PURL-keyed merge implementation (Decision 2)
- `conflict.rs` — `SCANNER_AUTHORITATIVE_FIELDS` + `DEVELOPER_AUTHORITATIVE_FIELDS` constants + `resolve_component()` (Decision 3)
- `annotation.rs` — helpers to stamp the three new `mikebom:*` annotation keys (Decision 5 + Decision 6 + FR-011)

**Rationale**:
- **Sibling to `scan_fs/`, `binding/`, `generate/`**: the placement reflects that supplement is conceptually parallel to filesystem scanning (a separate input source), not nested inside it.
- **Five small files vs. one large file**: each file's purpose is well-bounded and reviewable as a single unit. The total module size (~550 LoC) is comparable to existing per-feature modules (`binding/` is ~600 LoC; `binary/source_binding/` is ~400 LoC).
- **Test coverage**: each file has its own `#[cfg(test)] mod tests` block for unit-level coverage; integration tests live in `mikebom-cli/tests/supplement_cdx_integration.rs` per the milestone-113 / 118 precedent.

**Alternatives considered**:
- **One file `mikebom-cli/src/supplement.rs`** — rejected: ~550 LoC in one file becomes hard to navigate + invites unrelated concerns to bleed together.
- **Nest under `scan_fs/`** — rejected: incorrect mental model (supplement is operator-supplied, not filesystem-derived).

## Decision 8 — Constitution Principle V audit for three new annotation keys

**Decision**: Three new rows are added to `docs/reference/sbom-format-mapping.md`, each carrying a Principle V audit narrative:

### C65 — `mikebom:source-tier = "declared"` (extension to existing C5 row)

**Native-field audit**: CDX 1.6 has no native field expressing "this component was operator-declared, not scanner-observed." Closest neighbor is `evidence.identity[].methods[]` which carries identification methods but not source-tier provenance. SPDX 2.3 `Package.primaryPackagePurpose` is a category taxonomy, not a source-tier marker. SPDX 3.0.1's evidence-profile model would express this via `software_packagedBy`-style relationships in a future profile, but the 3.0.1 stable schema doesn't have a stable field for it.

**Decision**: Add `"declared"` as a new permitted value on the existing C5 `mikebom:source-tier` annotation. This is a value-set extension, not a new annotation key — preserves backwards compatibility for consumers reading C5 today.

### C66 — `mikebom:supplement-cdx` (envelope-level, new C-row)

**Native-field audit**: NO native field expressing "this SBOM merged in a supplement file with hash X" exists in CDX 1.6, SPDX 2.3, or SPDX 3.0.1. CDX `metadata.tools.components[]` lists tools that contributed to the SBOM but not external supplement files. SPDX 2.3 `creationInfo.creatorComment` is free-form prose, not structured provenance. SPDX 3.0.1's typed-tool model doesn't cover operator-supplied input files.

**Decision**: Add as a new envelope-level C-row. Per Principle X (Transparency) — operators reading the emitted SBOM MUST be able to detect that a supplement was used; the annotation provides this signal in a machine-parseable form.

### C67 — `mikebom:assertion-conflict` (per-component, new C-row)

**Native-field audit**: NO native field expressing "scanner observed X but operator declared Y; here's who won and why" exists in any of the three formats. CDX `evidence.identity` carries identification confidence + method, not conflict resolution. SPDX 2.3 `Annotation` is free-form prose. SPDX 3.0.1's evidence-profile model would express this via a future `conflictResolution` profile that doesn't exist in 3.0.1 stable.

**Decision**: Add as a new per-component C-row. Per Principle X — every conflict resolution decision MUST be auditable; the annotation provides the structured record so consumers can audit operator assertions against scanner findings.

**Rationale (all three)**:
- Each new key has a precedent C-row pattern: C5 (source-tier) for C65; C63 (exclude-path) for C66; C40 (component-role) for C67.
- All three follow the standing Principle V bullet 5 pattern: native field would be preferred, but none exists; the `mikebom:*` annotation is justified + documented in `docs/reference/sbom-format-mapping.md`.
- All three are EMISSION-GATED — present only when relevant (a supplement was used; a conflict occurred). Absence is the byte-identity preserved state for non-supplement scans (FR-013).

**Alternatives considered**:
- **Single combined `mikebom:supplement` annotation carrying all three semantics** — rejected: violates the per-key separation that makes parity-extractor catalogs maintainable.
- **Use existing C5 + C40 + C63 patterns without adding new keys** — rejected: the new semantics genuinely don't fit existing keys without conflating distinct concepts.
