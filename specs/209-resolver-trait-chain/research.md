# Research: Resolver Trait + Chain Refactor (m209)

**Date**: 2026-07-18
**Phase**: 0 (pre-design)
**Purpose**: Resolve technical unknowns before Phase 1 design. Every open decision from the plan surfaces gets a Decision / Rationale / Alternatives entry.

---

## R1: Async-in-sync mechanism (Spec FR-015)

**Decision**: Use native Rust `async fn` in traits (RPITIT — return-position `impl Trait` in trait), stable since Rust 1.75. Trait signature:

```rust
pub(crate) trait Resolver: Send + Sync {
    fn name(&self) -> &'static str;
    fn priority(&self) -> u32;
    fn technique(&self) -> ResolutionTechnique;
    fn confidence(&self) -> f32;
    fn handles(&self, input: &ResolveInput) -> bool;
    async fn resolve(
        &self,
        input: &ResolveInput<'_>,
        ctx: &ResolveContext<'_>,
    ) -> Result<Vec<ResolvedComponent>, ResolverError>;
}
```

Sync resolvers (Cargo/PyPI/npm/Golang/Maven/RubyGems/Deb/Path/Hostname) implement `async fn resolve` as `async fn` bodies that never `.await` — the compiler generates a no-op future that resolves immediately. Async resolvers (DepsDevHash) `.await` on the network call as normal. One trait shape covers both cases.

**Rationale**:
- Zero new Cargo dependencies (satisfies plan constraint + Constitution Principle VI).
- Idiomatic modern Rust; matches the direction the language is going (RPITIT stabilized specifically to obviate `async-trait` for most cases).
- The pipeline is already async (`ResolutionPipeline::resolve` at `pipeline.rs:65` is `async fn`), so no runtime-block hack is needed.
- Compile output is identical to what `async-trait` would produce (a `Pin<Box<dyn Future>>`-equivalent) but generated by the compiler, not a macro.
- The `handles(&self, input)` method lets the pipeline skip `.await`-invocation for resolvers that don't apply to the input type — micro-optimization + a natural filter step.

**Alternatives considered**:
- **`async-trait` crate**: rejected — adds a dep with no benefit over RPITIT on our MSRV.
- **Two traits (`SyncResolver` + `AsyncResolver`)**: rejected — complicates the chain iteration; every call site would need to branch on which trait to dispatch to. Defeats the "one uniform trait" spirit of FR-001.
- **Sync-only trait; async resolvers use `tokio::runtime::Handle::current().block_on(...)`**: rejected — deadlock risk if invoked from within an async context; violates tokio's don't-block-in-async discipline.
- **Manual `Pin<Box<dyn Future>>` return type**: rejected — verbose, error-prone, obsoleted by RPITIT.

## R2: Compile-time priority-uniqueness check mechanism (Spec Q2 / FR-017)

**Decision**: `const fn` uniqueness check over a `const RESOLVER_REGISTRY: &[(&str, u32)]` array, with a `const _: () = assert_registry_priorities_unique(RESOLVER_REGISTRY);` module-scope invocation. Rust's `const fn` supports slice iteration + `assert!` (with formatting via `const_format` — but we skip that dep and use a static concatenation of the two colliding names via `const fn` + `str::eq_ignore_ascii_case`-friendly primitives).

**Sketch**:

```rust
// resolve/resolver_chain.rs
pub(crate) const RESOLVER_REGISTRY: &[(&str, u32)] = &[
    ("cargo",             100),
    ("pypi",               99),
    ("npm",                98),
    ("golang",             97),
    ("maven",              96),
    ("rubygems",           95),
    ("deb",                94),
    ("deps_dev_hash",      90),
    ("path",               70),
    ("hostname_fallback",  40),
];

const fn assert_registry_priorities_unique(reg: &[(&str, u32)]) {
    let mut i = 0;
    while i < reg.len() {
        let mut j = i + 1;
        while j < reg.len() {
            if reg[i].1 == reg[j].1 {
                // Compile-time panic — cargo build fails with the resolver
                // names in the const-eval error message.
                panic!(
                    "resolver priority collision — two resolvers declared \
                     matching priorities; edit RESOLVER_REGISTRY at \
                     mikebom-cli/src/resolve/resolver_chain.rs to give \
                     each resolver a unique priority"
                );
            }
            j += 1;
        }
        i += 1;
    }
}

const _: () = assert_registry_priorities_unique(RESOLVER_REGISTRY);
```

Adding a resolver with a duplicate priority produces a `cargo build` error naming both the file and the panic message. This satisfies FR-017's compile-time requirement.

**Rationale**:
- Pure stable Rust; no proc macro; no new crates.
- `const fn` in stable Rust has supported `while` + `panic!` since ~1.79; workspace MSRV is well past that.
- The panic message identifies the file + the fix; discoverability satisfies the SC-006 "contributor doesn't need to read pipeline.rs" goal.
- Downside: the panic message can't currently include the two colliding resolver names dynamically (const-eval doesn't allow runtime string formatting). Mitigation: the sorted registry + the file-name pointer in the panic message make manual diagnosis trivial (5–10 seconds).

**Alternatives considered**:
- **`static_assertions` crate**: rejected — adds a dep for one macro; the `const fn` + `assert!` pattern above covers the use case with stdlib.
- **Proc macro generating the registry from an attribute**: rejected — overkill for 11 resolvers; makes the registration point harder to grep for.
- **Trait-associated `const PRIORITY: u32`**: acceptable and used INSIDE each resolver impl for local documentation, but the CROSS-resolver uniqueness check still needs a central table (or a very awkward macro dance across the trait system). The table approach is cleaner.
- **Runtime check with panic in `ResolverChain::new()`**: rejected per Q2 — the user explicitly picked compile-time.

## R3: Chain iteration shape — one chain vs. two (per-connection + per-file-op)

**Decision**: Single chain iterated per input event. Introduce a `ResolveInput` enum:

```rust
pub(crate) enum ResolveInput<'a> {
    Connection(&'a Connection),
    FileOp(&'a FileAccessOperation),
}
```

Each resolver's `handles()` method filters to the input variants it cares about (URL-family resolvers handle `Connection` only; path resolver handles both — it can extract paths from either; hostname_fallback handles `Connection` only). The pipeline iterates over BOTH event streams (connections then file-ops), dispatching each input through the full chain, respecting `handles()` short-circuits.

**Rationale**:
- One uniform trait covers both current input types (matches FR-001 minimalism).
- `handles()` acts as a cheap filter — avoids the async-await cost when the resolver clearly doesn't apply.
- Extending to a new input type (e.g., a future `TlsHandshake` event) means adding one enum variant + updating each resolver's `handles()` — chain iteration itself is unchanged.
- Matches SBOMit's approach: their resolvers all take `[]FileInfo` and filter internally; the mikebom variant just adds a lightweight input-type tag.

**Alternatives considered**:
- **Two chains** (`ConnectionResolverChain`, `FileOpResolverChain`): rejected — doubles the registration point; adding a resolver that handles both event types (like path resolver could) means implementing two traits.
- **Trait per input type** (`ConnectionResolver`, `FileOpResolver`): rejected — same doubling problem; violates FR-001's "common Resolver interface" requirement.
- **Erase input type via `dyn Any`**: rejected — loses type safety; every resolver would `.downcast_ref()` at entry.

## R4: Per-connection first-match-wins semantic preservation

**Decision**: The current pipeline runs URL → hash → hostname per-connection with a `continue` after the first `Some(purl)` (see `pipeline.rs:188`). The chain refactor preserves this: `ResolverChain::run_on_connection(conn)` iterates resolvers in priority-descending order and short-circuits on the FIRST resolver that returns `Ok(components)` with `!components.is_empty()`. Ok(empty) means "clean no-match — try the next resolver" per Q1.

**Rationale**:
- SC-001 mandates byte-identical output; changing this semantic would silently produce different components for connections with multiple matching resolvers.
- The path-resolver iteration (per-file-op) is independent of the per-connection iteration; both use the same chain but with different `handles()`-filtered dispatches.

**Alternatives considered**:
- **Collect from all resolvers, dedupe by confidence**: rejected — different output shape (may surface additional components from lower-priority resolvers), violates SC-001. Save for a future milestone if desired.
- **First-Err bubbles up**: rejected — a single resolver's Err would skip subsequent resolvers, changing semantics. Per FR-013 the pipeline logs + skips; other resolvers still run.

## R5: `catch_unwind` around each resolver invocation

**Decision**: Wrap each `resolver.resolve(...).await` in `std::panic::AssertUnwindSafe(...).catch_unwind()` at the pipeline layer. On panic, emit a WARN log naming the resolver + a synthesized "panic" failure kind, then continue with the next resolver — same semantics as `Err(...)` per FR-013.

`AssertUnwindSafe` is safe here because resolvers are stateless (FR-014) — the "unsafe" part of `catch_unwind` is when a panicking function might have left shared mutable state in an inconsistent condition; stateless resolvers can't leave inconsistent state behind.

**Rationale**:
- FR-013 mandates panic-catch; the async equivalent is `FutureExt::catch_unwind` from `futures-util`. `futures-util` is transitively in the tree (via `tokio` internal deps) but not a direct dep. Options: (a) add `futures-util` as a direct dep, (b) use `tokio::task::spawn` + `.await` on the join handle (spawn converts panics into `JoinError`), (c) implement `catch_unwind` manually with `AssertUnwindSafe`.
- **Chosen**: option (b) — `tokio::task::spawn(async move { resolver.resolve(...).await })` + `.await` on the `JoinHandle`. A `JoinHandle::Err(join_err)` where `join_err.is_panic()` gives us the panic-signal without a new dep. Runtime overhead is a task-spawn per resolver invocation (~100 ns on modern tokio); well within the SC-004 5 % ceiling.

**Alternatives considered**:
- **Add `futures-util` direct dep**: rejected — zero-new-deps discipline.
- **Manual `catch_unwind` on `Future::poll`**: rejected — requires `Pin` + `poll` primitives at the pipeline layer; error-prone.
- **Skip panic-catch, rely on Principle IV's no-panic discipline**: rejected — FR-013 explicitly requires it as defense-in-depth (unrecoverable programmer errors escape Principle IV's contract).

## R6: How to preserve the "reference pipeline" for the SC-001 byte-identity harness

**Decision**: The pre-refactor `pipeline.rs::resolve` implementation is preserved as a `#[cfg(test)]`-gated module at `mikebom-cli/src/resolve/pipeline_legacy_reference.rs` (with a comment naming it as the m209 SC-001 oracle, scheduled for deletion in a follow-up milestone once byte-identity is proven durable). The byte-identity harness at `tests/resolver_chain_byte_identity.rs` calls both the legacy oracle and the new chain against the same fixture corpus and asserts identity.

**Rationale**:
- Byte-identity oracle needs to be in-tree so the test is reproducible across PRs.
- `#[cfg(test)]` gating keeps the legacy code out of the shipped binary.
- The oracle module can be deleted in a follow-up milestone (m209b?) once we've observed multiple releases without a byte-identity regression.

**Alternatives considered**:
- **Static reference JSON output**: rejected — regenerating goldens becomes ambiguous (which pipeline generated them?). Live-comparison against the actual legacy code is more trustworthy.
- **Delete legacy immediately**: rejected — loses the byte-identity oracle; any future regression can't be diagnosed against the reference implementation.

## R7: SC-004 benchmark corpus definition

**Decision**: The perf baseline is measured against the m083 knative-func audit fixture at `mikebom-cli/tests/fixtures/transitive_parity/cargo/` (already vendored; used by m083 + m208 as a substantial real-world scan target). The baseline is captured ONCE from the pre-refactor pipeline, stored as `mikebom-cli/tests/fixtures/resolver_chain/perf_baseline.json` (containing `{"baseline_wallclock_ms": N, "captured_at_commit": "<sha>", "captured_at_date": "..."}`), and locked. The perf test asserts post-refactor wall-clock ≤ 1.05 × baseline.

**Rationale**:
- m083 fixture is already committed + provides realistic diversity (multiple ecosystems, ~500 connections, ~2000 file-ops).
- Locking the baseline at commit time prevents perf-drift creep from silently absorbing regressions.
- The `#[ignore]`-gate keeps the perf test out of the default pre-PR run (matches m094 convention); operators can opt in.

**Alternatives considered**:
- **Synthetic microbenchmark**: rejected — perf on realistic inputs matters; synthetic benches don't capture cache-effect + branch-predictor realities.
- **Multi-fixture perf suite**: rejected — one representative fixture is sufficient to catch 5 %+ regressions; more fixtures = more maintenance without proportional signal.
- **Absolute wall-clock threshold (e.g., "must run in under 5 s")**: rejected — hardware-dependent; the ≤ 105 % relative check is portable across CI runners.

## R8: Compile-fail test for the priority collision without adding `trybuild`

**Decision**: Implement the compile-fail test as a small `build.rs`-driven fixture. `tests/resolver_priority_collision.rs` invokes `cargo build --manifest-path tests/fixtures/resolver_collision_fixture/Cargo.toml` as a subprocess and asserts non-zero exit + stderr contains the panic message from R2. The fixture crate is a minimal `mikebom-cli`-clone that adds a duplicate-priority resolver.

**Rationale**:
- Avoids adding `trybuild` (zero-new-deps).
- Subprocess-invocation pattern already used elsewhere (m053 `git describe`, m055 `go mod graph`, m173 cache-warmer).
- Test is slower (~2 s to build the fixture) but only runs when this file changes; not a pre-PR bottleneck.

**Alternatives considered**:
- **Add `trybuild` as dev-dep**: rejected per zero-new-deps.
- **Skip the compile-fail test; rely on eyeballing during PR review**: rejected — FR-017 says the mechanism must be verified; a test locks the guarantee.
- **Doc-test with `compile_fail`**: candidate for a simpler approach — a `///```compile_fail` doctest on the `RESOLVER_REGISTRY` const referencing a hypothetical collision. Merit: no subprocess; no fixture crate. Demerit: doctests are less discoverable + the panic message check is harder. **Decision**: try the doctest approach first; fall back to the subprocess-fixture if the doctest can't verify the panic message.
