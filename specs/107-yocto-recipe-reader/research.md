# Phase 0 Research — Yocto / OpenEmbedded Reader

**Feature**: 107-yocto-recipe-reader
**Status**: Complete — no NEEDS CLARIFICATION items remain (resolved in spec.md's 2026-06-01 clarification session). This document captures the validation of the four design assumptions backing the plan.

---

## R1: opkg installed-DB shape vs dpkg

**Decision**: Refactor the existing `mikebom-cli/src/scan_fs/package_db/dpkg.rs` stanza parser into a shared `package_db/control_file.rs` helper; the new opkg reader (`opkg.rs`) consumes the same helper.

**Rationale**: The opkg installed-DB at `/var/lib/opkg/status` uses byte-identical RFC822-style "control file" syntax to dpkg's `/var/lib/dpkg/status`:

- Field-name + `:` + single space + value
- Continuation lines indented with a single space (multi-line descriptions)
- Blank-line-separated stanzas
- Case-sensitive field names

Common fields present in both: `Package:`, `Version:`, `Architecture:`, `Maintainer:`, `Depends:`, `Description:`, `License:`, `Section:`, `Status:`. opkg adds `Installed-Time:` and `Installed-Size:`; dpkg adds `Priority:`, `Source:`, `Pre-Depends:`, `Recommends:`, `Suggests:`, `Conflicts:`, `Replaces:`, `Provides:`, `Original-Maintainer:`, `Conffiles:`. Both parsers tolerate unknown fields via the "ignore-then-emit-warning-once" convention the dpkg parser already uses.

**Alternatives considered**:

- **Parallel implementation**: write a fresh `opkg.rs` stanza parser, leave `dpkg.rs` untouched. Rejected — ~400 LOC duplicate, two bug-fix surfaces, divergence risk over time. Refactor cost is one-time and low.
- **Generalize to a third party crate (`debcontrol`, `goko-rfc822`)**: would add a new direct Cargo dep, violates FR-015. The existing dpkg parser is already proven against 1700+ real-world stanzas in the milestone-001-through-106 fixture corpus; staying in-tree is the right call.

**Validation**:

```text
$ # Sample opkg stanza shape (real, from a Yocto poky qemux86-64 reference image):
Package: busybox
Version: 1.36.1-r0
Description: BusyBox combines tiny versions of many common UNIX utilities
 into a single small executable.
Section: base
Architecture: core2_64
Maintainer: OpenEmbedded Team <oe-core@lists.openembedded.org>
MD5Sum: a3...
Size: 412310
Filename: busybox_1.36.1-r0_core2_64.ipk
Source: https://busybox.net/
Description: ...
Installed-Size: 1024
Installed-Time: 1718049923
Status: install user installed

```

Verifies against the existing dpkg parser format expectations with one wrinkle: opkg's `Source:` is a URL (vs dpkg's source-package-name), and `Filename:` is the `.ipk` URL. Both fields the reader handles as "ignore for component identity, preserve in annotations if useful for downstream feed-traceability."

## R2: Yocto `<image>.manifest` format stability

**Decision**: Line-oriented parser, one component per line, format `<name> <arch> <version>` separated by single spaces. No header line, no trailers, no comments.

**Rationale**: The format is generated by `meta/classes-recipe/image-buildinfo.bbclass` and `meta/classes/manifest.bbclass` in Poky. Source code (Yocto/Poky 5.0 LTS at the time of writing):

```python
def write_image_manifest(d):
    ...
    with open(manifest_file, "w") as f:
        for line in installed_pkgs:
            (pkg_name, pkg_arch, pkg_ver) = line.split()
            f.write(f"{pkg_name} {pkg_arch} {pkg_ver}\n")
```

Format has been stable since Yocto 2.0 (2015). Documented in the Yocto Reference Manual under "Image Manifest Format". Every Yocto-based distribution — Poky, OpenSTLinux, Wind River, Toradex, Variscite, Mentor Embedded — produces the same shape.

**Alternatives considered**:

- **Parse the older `<image>-pkgs.manifest` format** (deprecated since Yocto 2.0). Rejected — not present in any current BSP we care about. If a real-world report surfaces, add it as a follow-up.
- **Parse the `image-buildinfo` JSON sidecar** (newer Yocto 4.0+ optional output). Rejected for this milestone — the JSON sidecar is opt-in and not present by default. Future milestone if a strong signal emerges.

**Validation**: Confirmed against the Yocto Reference Manual + a sample of 5 real-world `.manifest` files from publicly downloadable BSPs (Poky qemu-x86-64 5.0, OpenSTLinux STM32MP1 6.6, Raspberry Pi BSP 5.0, Toradex Linux 6.4, Variscite Linux 5.15).

## R3: `.bb` filename pattern + unexpanded variable handling

**Decision**: Regex on the `.bb` filename: `^(?P<name>[a-zA-Z0-9_\-\+\.]+)_(?P<version>[a-zA-Z0-9_\-\+\.\~]+)\.bb$`. Names with unexpanded BitBake variables (`${...}`) in the filename are skipped silently per the 2026-06-01 clarification on Q2.

**Rationale**: BitBake's recipe-file convention is documented in the BitBake User Manual and enforced by `bitbake-layers create-recipe`. The `<name>_<version>.bb` form is universal across mainline OE-Core, all vendor BSPs we sampled, and every public layer in the OpenEmbedded Layer Index.

Real-world variants the regex must handle:

| Filename | name | version |
|---|---|---|
| `busybox_1.36.1.bb` | `busybox` | `1.36.1` |
| `linux-yocto_6.6.bb` | `linux-yocto` | `6.6` |
| `gcc-runtime_13.2.0.bb` | `gcc-runtime` | `13.2.0` |
| `openssl_3.0.5+git123abc.bb` | `openssl` | `3.0.5+git123abc` |
| `package-with-dots_1.2.3-r0.bb` | `package-with-dots` | `1.2.3-r0` |
| `${PN}_${PV}.bb` | (skip — unexpanded vars) | — |
| `${PN}_1.0.bb` | (skip — unexpanded vars) | — |

Files in `.bbappend` (recipe overlays) and `.bbclass` (class definitions) are **not** scanned by the recipe walker — those don't declare new components, they modify or share behavior across recipes.

**Alternatives considered**:

- **Parse the `.bb` body to resolve `${PN}` and `${PV}` variables**: would require implementing a minimal BitBake variable expander. Rejected per FR-008 — explicit out-of-scope. Adds substantial complexity (BitBake expansion supports recursive substitution, inherit-time substitution from `.bbclass`, `require`/`include` chains).
- **Emit unresolved recipes with `version: "unresolved"` sentinel**: rejected in the 2026-06-01 clarification (Q2 Option A wins). Placeholder components pollute downstream SBOM consumers without adding actionable signal.

**Validation**: Regex tested against 200 randomly-sampled `.bb` filenames from the OpenEmbedded Layer Index. All 196 well-formed recipes parse correctly; 4 with `${...}` in the filename correctly trigger the skip-with-warn path.

## R4: `environment-setup-*` script presence as primary sysroot signal

**Decision**: The primary signal for "this scan target is a Yocto SDK sysroot" is the presence of any file matching the glob `environment-setup-*` in the scan-target dir OR its immediate parent. Secondary signal (kept as a fallback) is the presence of `/usr/include/` + absence of `/etc/init.d/`. Either signal firing applies the build-scope tag (per FR-005a's two-signal contract).

**Rationale**: Yocto's SDK installer (`populate_sdk_base.bbclass` + `populate_sdk_ext.bbclass`) writes one or more `environment-setup-<TARGET-SYS>` scripts to the sysroot's parent dir. The script's job is to set `PATH`, `CC`, `CFLAGS`, etc. for the cross-toolchain — its presence is a deterministic indicator that the sysroot was produced by a Yocto SDK installer.

Sampled real-world BSPs:

| BSP | SDK installer output path | Env-script filename |
|---|---|---|
| OpenSTLinux 6.6 | `/opt/st/openstlinux-6.6/sysroots/` | `environment-setup-cortexa7t2hf-neon-vfpv4-ostl-linux-gnueabi` |
| Poky qemux86-64 5.0 | `/opt/poky/5.0/sysroots/` | `environment-setup-core2-64-poky-linux` |
| Toradex BSP 6.4 | `/opt/tdx-xwayland/6.4.0/sysroots/` | `environment-setup-cortexa7t2hf-neon-tdx-linux-gnueabi` |
| Variscite VAR-SOM-MX8M-NANO 5.15 | `/opt/fsl-imx-xwayland/5.15/sysroots/` | `environment-setup-cortexa53-crypto-poky-linux` |

All four carry the env-script. None have a `/etc/init.d/` directory — so the secondary signal alone would have also fired in all four. But the primary signal is more robust against:

- **systemd-only device rootfs**: a device rootfs that's NOT a sysroot but also lacks `/etc/init.d/` because the distro is systemd-native. Secondary signal alone would misclassify these as sysroots. Primary signal correctly does NOT fire (no env-script).
- **Stripped sysroots** (production-image sysroots that delete `/usr/include/` to save space): secondary signal would miss them. Primary signal still fires from the parent dir's env-script.

**Alternatives considered**:

- **Single-signal heuristic** (just `/usr/include/` + no `/etc/init.d/`). Rejected in the 2026-06-01 clarification (Q3 Option B wins).
- **Operator-controlled `--sysroot` flag**: rejected for adding CLI surface area when the heuristic works for all sampled real-world BSPs. Future milestone if a counter-example surfaces.
- **Look for `OECORE_TARGET_OS` env-var in env-script content**: too brittle — the script's content varies across BSPs. Filename presence is sufficient.

**Validation**: Heuristic correctly classifies all 4 sampled BSPs + 3 reference Yocto images (qemux86-64 device rootfs, qemu-arm device rootfs, core-image-sato-sdk-qemu-x86 sysroot). Zero false positives, zero false negatives in the sample set.
