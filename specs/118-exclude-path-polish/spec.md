# Feature Specification: Milestone 113 `--exclude-path` polish bundle

**Feature Branch**: `118-exclude-path-polish`
**Created**: 2026-06-13
**Status**: Draft
**Input**: Issue #343 follow-up to milestone 113. The milestone-113 `--exclude-path` flag (PR #336) shipped with five integration tests proving the feature works end-to-end via the cargo ecosystem; the underlying walker plumbing is uniform across every ecosystem (cargo / maven / gem / pip / npm / gradle / nuget / yocto / golang-source / go-binary) — particularly after milestone 114's `safe_walk` migration unified all walker code through one shared helper. The deferred polish tasks (T020/T021/T023/T026/T027/T031/T032/T033/T034 from `specs/113-exclude-path-flag/tasks.md`) add per-ecosystem regression coverage, edge-case coverage (multi-pattern union, cross-platform separator), discoverability (CLI help test + per-ecosystem docs cross-links), observability (tracing summaries), and a performance regression check. This feature bundles them into one PR.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A maintainer can trust `--exclude-path` works uniformly across every ecosystem, not just cargo (Priority: P1)

A maintainer maintains the milestone-113 `--exclude-path` flag and watches as new contributors land changes in ecosystem readers (cargo, maven, gem, pip, npm, gradle, nuget, yocto, golang source, go binary). Without per-ecosystem regression tests, a future refactor in any one of those readers — say, a future Go reader change that bypasses the shared walker helper — could silently break `--exclude-path` for THAT ecosystem while the existing cargo-tier MVP tests stay green. The maintainer needs to evaluate every incoming PR against per-ecosystem coverage so this can't slip silently. After this feature, every ecosystem covered by the milestone-113 walker plumbing has at least one integration test asserting `--exclude-path` suppresses fixtures correctly for that ecosystem, AND the dependency-edge suppression / scan-root-excluded edge cases are explicitly covered.

**Why this priority**: This is the milestone-113 promise the MVP tests deferred: "the underlying walker plumbing is uniform across every ecosystem; per-walker tests are LESS valuable as bug-finders post-milestone-114, but MORE valuable as ecosystem-specific regression coverage." Without these, a future regression in any individual ecosystem reader silently breaks `--exclude-path` for that ecosystem until an operator hits the bug in production. P1 because every ecosystem's regression coverage is a foundation for trusting the flag across the polyglot operator surface.

**Independent Test**: For each of the four uncovered tests (golang source + go binary + dependency-edge suppression + scan-root excluded), construct a per-test fixture (or use an existing one) and an integration test that scans the fixture both with and without the relevant `--exclude-path` entries, then asserts the expected component set differs as predicted. The fixtures are vendored locally (NOT in the milestone-090 external fixture repo — these are synthetic test scaffolds, not real-world corpora).

**Acceptance Scenarios**:

1. **Given** a Go project laid out under `tests/fixtures/golang-source/` (deliberately NOT under `testdata/` or `_`-prefix dirs so the Go-tool unconditional skip from milestone 113 doesn't fire), **When** the operator scans with `--exclude-path 'tests/fixtures'`, **Then** zero `pkg:golang/...` components from that fixture appear in the emitted SBOM.
2. **Given** a fixture containing a pre-built Go binary at `tests/fixtures/go-binary/bin/foo`, **When** the operator scans with `--exclude-path 'tests/fixtures'`, **Then** zero `pkg:generic/foo` components from that binary's classification appear in the emitted SBOM (FR-013 binary-tier coverage).
3. **Given** a multi-ecosystem fixture where ecosystem A's component depends on ecosystem B's component, **When** the operator scans with `--exclude-path` suppressing the directory housing ecosystem B's component, **Then** the dependency edges pointing AT the suppressed component are dropped (no dangling `DEPENDS_ON` references), matching the milestone-114 milestone-115 dependency-graph emission contract.
4. **Given** a scan where the operator passes `--exclude-path '<scan-root>'` (the project root itself), **When** the scan runs, **Then** the SBOM emits ONLY the metadata.component (mikebom's own self-description) and no other components — every ecosystem walker correctly short-circuits when the scan root is excluded.

### User Story 2 — An operator using complex pattern combinations gets the expected behavior (Priority: P2)

An operator wants to exclude multiple distinct directory shapes simultaneously — for example, `**/testdata` AND `**/_archive` — to clean up SBOM noise from both test fixtures and archived sub-projects in one scan. Separately, an operator on a Windows-style filesystem (or one mistakenly using backslash separators in their `--exclude-path` argument) wants the same suppression behavior they'd get with forward-slash separators. After this feature, both cases work and have integration tests pinning the behavior.

**Why this priority**: The milestone-113 MVP tests cover ONE pattern entry at a time, with forward-slash separators only. Real operators routinely combine patterns and routinely pass paths that happen to be backslash-separated (Windows env, or simply copy-paste from a Windows-side tool). Both must work for the feature to be production-trustworthy across the polyglot operator surface.

**Independent Test**: Two per-test fixtures with the expected pattern combinations and separator forms; integration tests assert the expected suppression behavior.

**Acceptance Scenarios**:

1. **Given** a fixture with both a `services/<X>/testdata/<ecosystem>/<manifest>` subtree AND a `services/<X>/_archive/<ecosystem>/<manifest>` subtree, **When** the operator scans with `--exclude-path '**/testdata' --exclude-path '**/_archive'`, **Then** components from BOTH subtree shapes are suppressed in the same scan.
2. **Given** an `--exclude-path 'tests\fixtures'` argument (backslash-separated literal entry), **When** the operator scans on a Linux or macOS host, **Then** the same directories are suppressed as when passing `tests/fixtures` (FR-009 cross-platform separator normalization).

### User Story 3 — A new operator can discover and verify `--exclude-path` without reading source (Priority: P3)

A new operator running `mikebom sbom scan --help` sees `--exclude-path` listed with a one-line description and a pointer to the user-guide CLI reference. They read the cli-reference entry, which contains a fully-worked example plus a troubleshooting matrix; they cross-link from their ecosystem's section in `docs/ecosystems.md` to confirm `--exclude-path` is the right tool for their use case. After running the flag, they see a `tracing::info!` summary at scan end stating how many exclusion entries were applied and how many directories were suppressed; they can pass `MIKEBOM_LOG=debug` to see per-walker debug lines if they need to drill down. Finally, they want assurance that adding `--exclude-path` to their CI scans doesn't introduce a performance regression — they run an opt-in perf benchmark and confirm the with-flag wall time is ≤1.10× no-flag.

**Why this priority**: Discoverability + observability + performance assurance are the polish layer that separates a feature operators TRUST from one operators avoid. Without these, the feature exists but operators don't reach for it. P3 because the feature is functionally correct after PR #336; this story makes it ergonomic.

**Independent Test**: Three independent tests: (a) `mikebom scan --help` stdout contains `--exclude-path` AND the user-guide pointer; (b) every ecosystem section in `docs/ecosystems.md` carries a cross-link to the `--exclude-path` cli-reference entry; (c) the opt-in perf benchmark `exclude_path_overhead_within_budget` (ignored by default) scans a polyglot fixture with and without `--exclude-path '**/testdata'` and asserts with-flag wall time ≤1.10× no-flag.

**Acceptance Scenarios**:

1. **Given** the post-feature mikebom binary, **When** the operator runs `mikebom sbom scan --help`, **Then** the help output lists `--exclude-path` with a one-line description AND a pointer to `docs/user-guide/cli-reference.md`.
2. **Given** any ecosystem's section in `docs/ecosystems.md` (cargo, maven, npm, …), **When** an operator reads it, **Then** they find a cross-link to the `--exclude-path` CLI-reference entry.
3. **Given** a scan that uses `--exclude-path` with one or more entries, **When** the scan completes, **Then** a `tracing::info!` line at scan end states "exclude-path: applied N entries, suppressed M directories" (or equivalent shape matching the milestone-112 FR-013 stderr-summary pattern); per-walker `tracing::debug!` lines emit during the scan for operators who run with `MIKEBOM_LOG=debug`.
4. **Given** the opt-in `cargo +stable test --test exclude_path_perf_overhead -- --ignored` invocation, **When** the test runs on a polyglot fixture, **Then** with-flag wall time ≤ 1.10 × no-flag wall time (within the SC-003 budget from milestone 113).

### Edge Cases

- What happens when the per-walker tracing summary fires on a scan that used NO `--exclude-path` entries? The summary line is OMITTED entirely (absence = "no exclusions in effect," same byte-identity preservation rule from milestone 113 FR-003). The summary fires only when ≥1 entry was supplied. Matches the milestone-113 envelope-annotation emission gating.
- What happens when a per-ecosystem integration test's fixture coincidentally matches an existing exclusion shape (e.g., a directory named `_archive` in a fixture meant to test something unrelated)? Each integration test's fixture path is chosen to NOT match the default Go-tool skip shapes (`testdata/` and `_`-prefix dirs) UNLESS the test specifically intends to exercise those code paths. Test authors must verify their fixture doesn't accidentally trip the unconditional skip behavior from milestone 113.
- What happens when the perf benchmark runs on a CI macOS lane where thermal/scheduler noise routinely produces 14–22% measured-reduction false-positives (per `specs/094-deflake-perf-tests/`)? The perf benchmark is `#[ignore]`d in the default `cargo test` invocation. It runs only on explicit opt-in via `--ignored`. CI does NOT run it on per-PR lanes. Same posture as the existing `triple_format_perf.rs` / `dual_format_perf.rs` benchmarks; matches the milestone-094 CI-perf-gate rationale.
- What happens when the cross-platform separator normalization test runs on Windows itself? The test specifies that both forward-slash and backslash forms normalize identically; the inverse direction (forward-slash literal on Windows) doesn't need a separate skip-on-Windows test because the same code path handles both forms. The integration test's assertions hold on every supported host.
- What happens when an ecosystem reader is added in a future PR but its docs section in `docs/ecosystems.md` doesn't carry the `--exclude-path` cross-link? The current ecosystem set is fixed at this feature's open-time; future ecosystems are responsible for their own docs cross-link. The discoverability promise is "every ecosystem section that exists at PR-ship time has a cross-link" — not "every future ecosystem section will."
- What happens when the help-text test runs against a `mikebom sbom scan --help` whose flag description text changes for an unrelated reason? The test asserts the SUBSTRING `--exclude-path` appears in stdout AND that the substring `docs/user-guide/cli-reference.md` (or equivalent pointer) appears in the help text. The test does NOT assert exact wording — operators can rewrite the description without breaking the test.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The mikebom test suite MUST include a per-ecosystem integration test that scans a vendored Go-source fixture with `--exclude-path` and asserts zero `pkg:golang/...` components from that fixture appear in the emitted SBOM. The fixture path is chosen to NOT match the Go-tool default skip shapes (`testdata/` and `_`-prefix dirs) so the test exercises `--exclude-path` exclusively rather than the Go-tool skip.
- **FR-002**: The mikebom test suite MUST include an integration test that scans a vendored fixture containing a pre-built Go binary at `tests/fixtures/go-binary/bin/foo` with `--exclude-path` and asserts zero `pkg:generic/foo` components from that binary's classification appear in the emitted SBOM.
- **FR-003**: The mikebom test suite MUST include an integration test asserting that when `--exclude-path` suppresses a directory housing a component that other components depend on, the dependency edges pointing AT the suppressed component are dropped (no dangling `DEPENDS_ON` references in the emitted SBOM).
- **FR-004**: The mikebom test suite MUST include an integration test asserting that when the operator passes `--exclude-path '<scan-root>'` (the project root itself), the SBOM emits ONLY the metadata.component (mikebom's own self-description) and no other components.
- **FR-005**: The mikebom test suite MUST include an integration test asserting that two distinct `--exclude-path` pattern entries in the same scan (e.g., `--exclude-path '**/testdata' --exclude-path '**/_archive'`) suppress BOTH subtree shapes correctly — the union of the two patterns matches both subtrees.
- **FR-006**: The mikebom test suite MUST include an integration test asserting that an `--exclude-path` argument with backslash separators (`tests\fixtures`) suppresses the same directories as the forward-slash form (`tests/fixtures`) on the test's host. The test does NOT require running on a Windows host; both forms normalize identically in the parser, so the assertion holds on any supported host.
- **FR-007**: `docs/ecosystems.md` MUST contain ONE consolidated `--exclude-path` reference section (e.g., `## Excluding paths from a scan`) that carries the canonical operator-facing explanation cross-linking to `docs/user-guide/cli-reference.md` § `--exclude-path`. Each ecosystem section (cargo, maven, gem, pip, npm, gradle, nuget, yocto, golang) MUST carry a short pointer line — one short sentence + a section anchor link — directing operators to that consolidated reference. Future updates to `--exclude-path` documentation touch ONE consolidated section rather than ten ecosystem-section duplicates.
- **FR-008**: The mikebom test suite MUST include a test asserting that `mikebom sbom scan --help` stdout contains the substring `--exclude-path` AND a substring identifying the user-guide CLI-reference entry (e.g., `docs/user-guide/cli-reference.md` or `cli-reference`). The test does NOT assert exact description wording, only the discoverability anchors.
- **FR-009**: Every per-ecosystem walker MUST emit a `tracing::debug!` line summarizing exclusion behavior during the scan (matching the milestone-114 `safe_walk`-centralized skip-event pattern). The `safe_walk` helper already emits these centrally; this requirement is satisfied by every ecosystem reader correctly delegating to `safe_walk` (a milestone-114 invariant, enforced by the walker-audit gate from milestone 115).
- **FR-010**: At scan end, when `--exclude-path` had ≥1 entry in effect, a `tracing::info!` summary line MUST emit stating "exclude-path: applied N entries, suppressed M directories" (or equivalent shape matching the milestone-112 FR-013 stderr-summary pattern). When zero entries were supplied, the summary line MUST be omitted entirely (absence = "no exclusions in effect"; preserves byte-identity for non-exclusion scans).
- **FR-011**: The mikebom test suite MUST include an opt-in (default-ignored via Rust's `#[ignore]` attribute) performance regression test asserting that scanning the existing `kusari-cli` polyglot fixture (already cached locally per milestone 090, pinned via `tests/fixtures.rev`) with `--exclude-path '**/testdata'` has wall time ≤1.10 × the no-flag baseline wall time, matching milestone 113's SC-003 budget. The test does NOT run in the default `cargo test` invocation; it runs only via explicit opt-in (e.g., `cargo +stable test --test exclude_path_perf_overhead -- --ignored`).
- **FR-012**: All new fixtures vendored by this feature MUST live IN-TREE under `mikebom-cli/tests/fixtures/exclude_path_polish/<sub-fixture>/` rather than in the milestone-090 external fixture-repo. These are synthetic test scaffolds, not real-world corpora; in-tree keeps them close to their tests and avoids forcing a fixture-repo bump.

### Key Entities

- **Per-ecosystem integration test**: A Rust integration test under `mikebom-cli/tests/exclude_path_polish_<ecosystem>.rs` (or appended to the existing `mikebom-cli/tests/exclude_path_integration.rs`) that exercises `--exclude-path` for one specific ecosystem's reader. Distinguished from the milestone-113 MVP tests by ecosystem scope.
- **Per-ecosystem fixture**: A vendored synthetic project under `mikebom-cli/tests/fixtures/exclude_path_polish/<ecosystem>/` providing the minimum manifest shape needed to exercise the ecosystem's reader. Multi-ecosystem fixtures (for FR-003 dependency-edge suppression) span multiple subdirectories under one parent.
- **CLI-help test**: An integration test that invokes the mikebom binary with `sbom scan --help` and parses stdout for the discoverability anchors (the `--exclude-path` flag name + the user-guide pointer).
- **Tracing summary**: A `tracing::info!` line emitted at scan end when ≥1 `--exclude-path` entry was in effect, mirroring the milestone-112 FR-013 stderr-summary pattern. Per-walker `tracing::debug!` lines emit during the scan via the milestone-114 `safe_walk` helper.
- **Perf-overhead test**: An opt-in (`#[ignore]`d) integration test that times scans of a polyglot fixture both with and without `--exclude-path '**/testdata'`, asserting the with-flag wall time is within the SC-003 budget.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: After this feature ships, every ecosystem covered by the milestone-113 walker plumbing has at least one integration test asserting `--exclude-path` works for that ecosystem. The current ecosystem set is: golang source (FR-001), go binary (FR-002), plus the dependency-edge edge case (FR-003) and the scan-root-excluded edge case (FR-004). Cargo / maven / gem / pip / npm coverage is already in place from milestone 113's MVP tests.
- **SC-002**: A maintainer reviewing a future PR that touches any ecosystem reader can verify the PR doesn't silently break `--exclude-path` for that ecosystem by running the per-ecosystem integration test (or the test suite catches the regression automatically on every PR via the default-on CI test runs).
- **SC-003**: A new operator running `mikebom sbom scan --help` discovers `--exclude-path` AND a pointer to the user-guide CLI-reference entry without reading source.
- **SC-004**: An operator reading any ecosystem section in `docs/ecosystems.md` finds a cross-link to the `--exclude-path` CLI-reference entry, so they don't have to search for the flag separately.
- **SC-005**: An operator inspecting a scan's stderr after running with `--exclude-path` sees the scan-end summary line stating how many entries applied and how many directories were suppressed. When running with `MIKEBOM_LOG=debug`, per-walker debug events emit during the scan.
- **SC-006**: The opt-in perf benchmark, when run by a maintainer, confirms the with-flag wall time is ≤1.10× no-flag wall time on a polyglot fixture. The benchmark does NOT run in default CI lanes (matches the milestone-094 thermal-noise rationale).

## Assumptions

- The fixtures are vendored in-tree at `mikebom-cli/tests/fixtures/exclude_path_polish/<ecosystem>/` rather than in the milestone-090 external fixture-repo per FR-012. They're synthetic; ~10–30 KB each. The milestone-090 fixture-repo cache is reserved for real-world corpora (e.g., the kusari-sandbox repo).
- The `--exclude-path` flag's CLI behavior is unchanged from milestone 113. This feature adds tests + docs + tracing; it does NOT modify the flag's parsing or matching logic. If a future change to `--exclude-path` semantics is needed, that's a separate milestone.
- The cross-platform separator normalization in FR-006 is already implemented in milestone 113's parser; this feature only adds the test asserting that behavior. If the test FAILS on PR-open, the test stays in the PR and the parser fix becomes part of this feature's scope.
- The opt-in perf benchmark (FR-011) follows the milestone-094 convention: `#[ignore]`d by default, runs only on explicit `--ignored` opt-in. It does NOT block CI on default-on lanes per the milestone-094 thermal-noise rationale.
- The tracing summary's exact format is at the implementer's discretion within the FR-010 envelope ("exclude-path: applied N entries, suppressed M directories" OR equivalent). Operators inspect via stderr; the format change is captured in the cli-reference docs if it deviates from this string.
- The per-ecosystem coverage scope is golang source + go binary + dependency-edge edge case + scan-root edge case (FR-001 through FR-004). Cargo / maven / gem / pip / npm / nuget / gradle / yocto ecosystems already have either explicit milestone-113 coverage OR delegate through `safe_walk` (milestone 114) such that their exclusion behavior is inherited from the shared helper. Per-ecosystem tests for THOSE ecosystems are deliberately OUT OF SCOPE for this feature; they may be added in a future PR if regression coverage proves insufficient.
- The docs cross-link from `docs/ecosystems.md` ecosystem sections is in scope; updating other docs (e.g., `docs/design-notes.md`, the README) is NOT in scope unless those docs already reference the `--exclude-path` flag.
- The single PR scope is ~1 day of focused work per the issue body. If task estimates expand mid-implementation, the perf-benchmark task (T034) may be deferred to a separate follow-up issue without blocking the rest of the polish bundle; the perf-benchmark is the least signal-dense of the nine tasks.
- The walker-audit gate (milestone 115/117) catches any new `fn walk*` introduced by this feature. The expected count of new walker functions: zero. All test-driven additions should use existing `safe_walk` integration or test helpers that don't match the audit pattern.
